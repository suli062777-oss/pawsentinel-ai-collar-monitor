# PawRoom Web 信息架构图理论审查

版本：v0.1  
日期：2026-07-07  
审查对象：`docs/assets/pawroom-web-ia-map-v0.2.svg`、`docs/assets/pawroom-web-ia-map-v0.2.png`、`docs/assets/pawroom-web-ia-map-v0.2.pdf`  
审查目的：用信息架构、用户体验概论和可用性启发式原则检查当前 PawRoom Web 原型 IA 是否合理。

---

## 1. 结论

当前信息架构图整体合理，可以作为 PawRoom Web 原型的主 IA 图继续使用。

综合判断：8.3 / 10。

它的主要优势是：

- 一级模块已收敛到 6 个，适合汇报和原型制作。
- 主路径清楚：宠物创建 -> 项圈数据 -> 看护首页 -> 桌面小世界 -> 今日回放创作。
- 核心心智没有跑偏：安全看护是主线，桌宠互动和 AI 创作是增强体验。
- 已经避免把硬件生产、手机端完整 App、支付系统等非 MVP 内容塞进主结构。
- 与竞品用户心智有关联：项圈、电量、位置、活动、安全提醒、趋势、锁屏通知都是用户能理解的概念。

主要风险是：

- “生命状态趋势”容易被用户误解为医疗能力，需要明确“非诊断提示”。
- “Paw Credits”属于产品内命名，第一次看图的用户不一定理解，主图里应优先用“AI 创作额度”。
- “手机状态预览”放在“桌面小世界”下面时容易显得层级不自然，应改成更抽象的“跨端状态预览”。
- 异常提醒路径如果只指向“提醒设置”，用户会不知道异常发生后要做什么，所以需要补“查看建议操作”。

以上 4 点已经同步修正到当前 SVG/PNG 图中。

## 2. 理论依据

### 2.1 信息架构基础

信息架构关注的是组织、结构、标签和可发现性。对 PawRoom 来说，IA 不应该只是列功能，而应该回答：

- 用户先看什么？
- 用户如何找到安全状态？
- 用户如何知道数据来自哪里？
- 用户如何从看护进入互动和回放？
- 哪些能力不应该占用一级入口？

当前 IA 把功能拆为“宠物创建、项圈与数据、看护首页、桌面小世界、今日回放与创作、我的与设置”，基本符合“以用户任务组织信息”的要求。

### 2.2 Garrett 用户体验五层模型

按《The Elements of User Experience》的经典思路，可以从战略层、范围层、结构层、框架层、表现层检查。

| 层级 | 检查点 | 当前判断 |
| --- | --- | --- |
| 战略层 | 产品是否服务核心目标 | 合理。核心目标是上班时低干扰看护宠物。 |
| 范围层 | 功能是否过多或过少 | 基本合理。P0 功能都覆盖，非核心功能被降级。 |
| 结构层 | 用户路径是否自然 | 合理。创建 -> 数据 -> 看护 -> 互动 -> 回放。 |
| 框架层 | 模块入口是否符合用户预期 | 基本合理。仍需在真实原型中突出“看护首页”为默认入口。 |
| 表现层 | 图是否易扫读 | 基本合理。参考图式树状结构清楚，节点数量可控。 |

### 2.3 Morville UX 蜂巢

Morville 的 UX Honeycomb 常用来检查一个体验是否有用、可用、可找、可信、可达、可欲、有价值。

| 维度 | PawRoom IA 判断 |
| --- | --- |
| Useful 有用 | 通过。主线是安全看护，不是单纯可爱桌宠。 |
| Usable 可用 | 基本通过。一级模块少，路径明确。 |
| Findable 可找 | 基本通过。关键任务有明确入口。 |
| Credible 可信 | 需要注意。必须区分项圈记录、用户补充、AI 演绎。 |
| Desirable 可欲 | 通过。桌面小世界和今日回放提供情绪价值。 |
| Accessible 可达 | IA 图层面无法完全判断，后续 UI 原型需要检查文字、对比度、按钮尺寸。 |
| Valuable 有价值 | 通过。安全看护支撑付费理由，AI 创作额度支撑增值模式。 |

### 2.4 Nielsen 可用性启发式

重点检查与 IA 直接相关的几条：

| 启发式 | 当前表现 | 修正/建议 |
| --- | --- | --- |
| 系统状态可见 | 看护首页有一句话状态、安全等级、当前区域 | 合理。原型中要把这几个放在首屏。 |
| 匹配真实世界 | 项圈、电量、活动、区域、提醒是用户熟悉词 | 合理。已把 Paw Credits 改为 AI 创作额度。 |
| 一致性与标准 | 结构接近智能项圈/宠物摄像头/桌面组件心智 | 合理。不要发明太多新名词。 |
| 错误预防 | 生命趋势和 AI 演绎存在误解风险 | 已补“非诊断提示”和数据边界。 |
| 识别优于回忆 | 树状结构让用户直接看见功能入口 | 合理。后续 UI 不要隐藏关键状态。 |
| 美学与极简 | 一级入口控制在 6 个 | 合理。旧版 Mermaid 信息太散，当前图更适合扫读。 |
| 帮助用户恢复 | 异常提醒需要给下一步动作 | 已补“查看建议操作”。 |

## 3. 当前 IA 的合理性判断

### 3.1 一级模块合理

当前 6 个一级模块是合适的：

1. 宠物创建：解决“这是我的宠物”的个体化入口。
2. 项圈与数据：解决可信度和数据来源。
3. 看护首页：解决安全状态的高频查看。
4. 桌面小世界：解决 PawRoom 的差异化体验。
5. 今日回放与创作：解决记忆留存和分享价值。
6. 我的与设置：承接配置、隐私、额度、边界说明。

这个结构比原先把“手机提醒”“Paw Credits”“演示控制”都放在一级更合理。用户第一次理解产品时，不应该同时面对太多入口。

### 3.2 用户路径合理

当前高频路径是：

> 宠物创建 -> 项圈与数据 -> 看护首页 -> 桌面小世界 -> 今日回放与创作

这条路径符合用户对产品价值的理解顺序：

1. 先拥有自己的宠物分身。
2. 再相信它由真实数据驱动。
3. 再看到安全状态。
4. 再进入低干扰陪伴。
5. 最后生成可分享的记忆内容。

这比直接从“桌宠互动”开始更好，因为 PawRoom 的付费基础是安全感，不是单纯娱乐。

### 3.3 异常提醒路径基本合理

当前异常提醒路径是：

> 项圈与数据 -> 看护首页 -> 跨端状态预览 -> 我的与设置

这条路径需要在实际 UI 中拆成两层：

- 当下处理：看护首页中查看原因和建议操作。
- 长期配置：到我的与设置中调整提醒偏好。

因此图里补了“查看建议操作”，避免用户以为异常只能去设置里处理。

### 3.4 数据边界处理合理

“项圈记录 / 用户补充 / AI 演绎”这个分组很关键，建议保留。

原因：

- PawRoom 有 AI 演绎能力，如果不区分来源，用户可能误以为 AI 文案就是宠物真实行为。
- 生命状态趋势有安全看护价值，但不能被表达成医疗诊断。
- 数据边界越清楚，产品可信度越高。

### 3.5 商业化入口处理合理

把 Paw Credits 从一级模块降级到“我的与设置”是合理的。

原因：

- 用户第一次看 IA 时，最关心的是宠物是否安全，不是额度。
- AI 创作额度是增值能力，不应该压过安全看护主线。
- “AI 创作额度”比“Paw Credits”更符合新用户理解，后续界面里可以再解释品牌名。

## 4. 已同步修正

已对 `pawroom-web-ia-map-v0.2.svg` 和导出的 PNG 做了以下修正：

1. “手机状态预览”改为“跨端状态预览”。
2. “Paw Credits”改为“AI 创作额度”。
3. 在生命状态趋势下补充“非诊断提示”。
4. 在看护首页快捷操作下补充“查看建议操作”。
5. 底部说明从“手机提醒与 Paw Credits”改为“跨端提醒与 AI 创作额度”。

这些修正主要解决：

- 新用户术语理解问题。
- 医疗误解风险。
- 异常提醒后的下一步动作问题。
- 跨端能力放在桌面模块下时的语义问题。

## 5. 仍建议后续验证

当前图可以进入原型制作，但正式汇报前建议做一次轻量验证。

### 5.1 Tree Testing

找 5 个目标用户或同学，让他们只看这张 IA 图，完成 4 个任务：

1. 想知道宠物现在是否安全，应该看哪里？
2. 想知道项圈数据是不是可信，应该看哪里？
3. 收到异常提醒后，应该从哪里看原因和建议？
4. 想生成今天宠物的一天小故事，应该去哪里？

通过标准：

- 每个任务 4/5 人能在 10 秒内指出正确模块。

### 5.2 Card Sorting

如果后续功能继续变多，再做一次开放式卡片分类，重点验证：

- “跨端状态预览”是否应该留在桌面小世界下。
- “AI 创作额度”是否应该留在我的与设置下。
- “今日回放与创作”是否需要拆成“今日回放”和“创作中心”。

当前 MVP 阶段不建议拆，拆了会让 IA 变重。

---

## 6. 最终判断

当前 IA 合理，适合继续用于 Web 原型设计。

建议在后续界面原型中严格保持三个原则：

1. 默认入口是“看护首页”，不是桌宠娱乐页。
2. 桌面小世界必须服务安全状态表达，而不是变成独立小游戏。
3. AI 创作必须区分“事实记录”和“AI 演绎”，避免可信度风险。

如果要继续优化，优先优化真实页面交互，不需要再大改 IA。
