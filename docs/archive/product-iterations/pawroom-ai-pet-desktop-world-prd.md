# PawRoom AI 宠物桌面小世界 PRD

版本：v0.1  
日期：2026-07-07  
文档类型：产品需求文档 / 开发 PRD  
产品阶段：机会验证 MVP  
目标周期：2 天完成可演示原型，2 周完成真实用户验证

## 1. 背景与问题定义

### 1.1 用户问题

养宠用户在上班、外出、出差时，会天然关心宠物在家里的状态。目前主流方式有三类：

- 照片/视频记录：适合回看，但不具备实时陪伴和互动感。
- 摄像头监控：能看真实画面，但体验偏监控，隐私压力高，且不够有趣。
- 智能项圈/追踪器：能提供位置、活动、睡眠、健康数据，但呈现方式偏工具化。

用户真正想获得的不是严肃监控，而是：

- 大概知道宠物在家是否正常。
- 上班时能在桌面上看到一个可爱的宠物小场景。
- 让现实宠物的照片、声音、活动数据，变成更有情绪价值的数字分身。
- 下班后能回看宠物今天的“小剧场”，而不是一堆冷冰冰的数据。

### 1.2 产品机会

现有强竞品集中在“数据采集层”：

- Tractive：GPS 实时定位、虚拟围栏、位置历史、活动/睡眠/健康指标。
- Fi：GPS、活动、睡眠，Series 3 Plus 增加叫声、舔舐、抓挠、吃喝等行为检测。
- PetPace：体温、脉搏、呼吸等生命体征监测，偏医疗健康。
- Invoxia Minitailz：GPS、活动、心率、呼吸等监测。
- Furbo / Petcube：宠物摄像头、远程查看、AI 提醒、互动。

但这些产品大多把数据展示为地图、图表、提醒或视频流。PawRoom 的机会不在硬件采集，而在“宠物状态的情绪化演绎层”：

> 读取或模拟智能项圈/宠物设备数据，把宠物在家的大概状态演绎成一个可互动、可观看、可分享的卡通桌面小世界。

### 1.3 核心判断

PawRoom 不应定位为“宠物实时监控工具”，而应定位为“AI 宠物桌面小世界”。

产品承诺不是 100% 精准复刻宠物行为，而是：

- 基于照片生成专属电子宠物形象。
- 基于活动/位置/事件数据推断大概状态。
- 用 AI 与动效把状态演绎成可爱的桌面场景。
- 通过今日小剧场、表情包、互动小游戏提高情绪价值和分享动力。

## 2. 产品目标

### 2.1 MVP 目标

在 2 天内完成一个可演示原型，证明以下体验闭环成立：

1. 用户上传宠物照片或选择示例宠物。
2. 系统生成卡通/像素风宠物分身。
3. 用户选择或导入一组宠物状态数据。
4. 桌面房间中展示宠物在狗窝、饭盆、门口、沙发等区域的动态状态。
5. 用户可以与宠物互动。
6. 系统生成“今日在家小剧场”或“今日冒险长图”。

### 2.2 2 周验证目标

验证该产品是否具备真实用户价值：

- 10-20 位养宠用户看完 Demo 后，是否理解产品价值。
- 至少 5 位用户愿意上传宠物照片体验生成。
- 至少 3 位用户愿意为高清小剧场、表情包或桌宠皮肤支付 9.9-29.9 元。
- 至少 3 位用户表示愿意在上班期间常驻打开桌宠窗口。
- 验证用户是否接受“大概真实 + 可爱演绎”，而不是要求严格实时监控。

### 2.3 不做的目标

MVP 不做以下事项：

- 不自研智能项链/项圈硬件。
- 不承诺接入所有第三方设备。
- 不做医疗诊断、健康风险判断、处方建议。
- 不做真实 24 小时视频监控。
- 不做大型宠物元宇宙或复杂养成游戏。

## 3. 目标用户与使用场景

### 3.1 核心用户

#### 用户 A：上班族养狗人

- 年龄：22-35 岁。
- 场景：白天上班，狗独自在家。
- 痛点：想知道狗是不是在睡觉、有没有焦虑、有没有等主人。
- 期待：桌面上有一个卡通小家，可以看到狗狗大概在做什么。

#### 用户 B：高分享欲养宠人

- 年龄：18-30 岁。
- 场景：喜欢在小红书、朋友圈、B站分享宠物日常。
- 痛点：普通照片视频同质化，想要更有趣的宠物内容。
- 期待：一键生成宠物小剧场、表情包、动态桌宠、漫画日记。

#### 用户 C：已有智能项圈/摄像头用户

- 年龄：25-40 岁。
- 场景：已购买 Tractive、Fi、Furbo、Petcube 等设备。
- 痛点：硬件 App 数据实用但不够好玩，缺少情绪陪伴。
- 期待：已有设备数据能变成更好看的桌面体验。

### 3.2 关键场景

#### 场景 1：上班时的桌面陪伴

用户打开电脑，PawRoom 以小窗口常驻桌面。小狗在卡通房间中睡觉、巡逻、靠近饭盆或等主人。用户不用打开摄像头，也能获得轻度安心和陪伴感。

#### 场景 2：下班后的今日小剧场

系统根据白天状态数据生成一段宠物在家的卡通历程：

- 09:30 在狗窝睡觉。
- 11:20 在门口徘徊。
- 13:10 饭盆附近活动。
- 15:40 沙发区域休息。
- 18:20 等主人回家。

输出为 15 秒短动画、四格漫画或长图。

#### 场景 3：用户主动添加趣味动作

设备数据无法判断所有细节，因此用户可以给宠物添加动作包：

- 撒娇。
- 打滚。
- 拆纸箱。
- 叼玩具。
- 趴窗。
- 等主人。

系统将动作包与真实状态结合，让桌宠更有表现力。

#### 场景 4：社交分享

用户一键导出：

- 今日小剧场短视频。
- 宠物表情包。
- 卡通房间截图。
- 宠物角色卡。

用于朋友圈、小红书、微信群分享。

## 4. 产品定位与差异化

### 4.1 一句话定位

PawRoom 是一个 AI 宠物桌面小世界：读取宠物照片、声音和智能项圈数据，把宠物在家的大概状态演绎成一个可互动、可观看、可分享的卡通桌宠场景。

### 4.2 与竞品的区别

| 类型 | 代表产品 | 用户获得 | PawRoom 差异化 |
| --- | --- | --- | --- |
| GPS 项圈 | Tractive、Fi | 位置、轨迹、活动数据 | 把轨迹和状态转成卡通家居小剧场 |
| 健康项圈 | PetPace、Invoxia | 生命体征、健康趋势 | 不做医疗判断，只做非医疗状态演绎 |
| 宠物摄像头 | Furbo、Petcube | 真实视频、远程互动 | 不依赖视频监控，降低隐私压力 |
| AI 宠物头像 | 各类图像生成工具 | 单张图片风格化 | 生成持续存在、可互动的宠物分身 |
| 桌宠工具 | 开源桌宠/Live2D | 通用虚拟角色 | 角色来自用户真实宠物和真实状态数据 |

### 4.3 核心卖点

1. 专属宠物分身：从现实照片生成，而非默认虚拟宠物。
2. 状态驱动演绎：根据活动、位置、事件数据改变桌宠行为。
3. 可爱但不伪装精准：明确是“大概真实 + AI 演绎”。
4. 适合常驻桌面：轻量、漂亮、不打扰。
5. 可分享输出：每日小剧场、表情包、宠物角色卡。

## 5. 用户故事

1. 作为养狗上班族，我想上传几张我家狗的照片，所以我可以生成一个像它的电子分身。
2. 作为养猫用户，我想选择像素风、手绘风、软萌 3D 风，所以我可以让宠物桌宠符合我的审美。
3. 作为用户，我想创建家里的卡通地图，所以我可以看到宠物在狗窝、饭盆、门口、沙发等区域活动。
4. 作为用户，我想导入或模拟智能项圈数据，所以桌宠可以根据宠物状态自动变化。
5. 作为用户，我想看到宠物当前处于睡觉、活跃、等待、吃饭、巡逻等状态，所以我能获得轻度安心。
6. 作为用户，我想知道这些状态不是严格实时监控，所以我不会误以为产品能替代摄像头或健康设备。
7. 作为用户，我想在上班时把桌宠窗口放在屏幕角落，所以我能随时看到卡通版的宠物在家生活。
8. 作为用户，我想点击宠物触发摸摸、喂食、叫它等动作，所以我能产生互动感。
9. 作为用户，我想给宠物添加趣味动作包，所以即使硬件数据不完整，桌宠也能有丰富表现。
10. 作为用户，我想在下班后生成“今日小剧场”，所以我能回看宠物今天的卡通化历程。
11. 作为用户，我想导出今日小剧场为图片或短视频，所以我可以发到社交平台。
12. 作为用户，我想生成宠物表情包，所以我可以在聊天中使用我家宠物的可爱表情。
13. 作为已有项圈用户，我想绑定第三方设备数据，所以我不需要额外购买硬件。
14. 作为没有项圈的用户，我想手动选择宠物状态，所以我也能体验产品核心玩法。
15. 作为隐私敏感用户，我想不使用摄像头，所以我可以在不暴露家中画面的情况下获得陪伴体验。
16. 作为产品运营者，我想看到用户生成、互动、导出、付费数据，所以我能判断体验是否成立。
17. 作为开发者，我想用统一的宠物状态协议接收不同数据源，所以后续接入设备时不需要重写桌宠逻辑。
18. 作为设计师，我想通过统一角色和动效规范控制风格，所以桌宠体验不会变成杂乱素材拼贴。
19. 作为测试人员，我想用模拟数据回放宠物一天的状态，所以我可以验证不同状态下的动画和场景表现。
20. 作为商业负责人，我想验证用户是否愿意为高清导出、皮肤、动作包或实体周边付费，所以我能判断商业化路径。

## 6. MVP 范围

### 6.1 MVP 必做功能

#### 6.1.1 宠物分身创建

输入：

- 宠物名称。
- 宠物类型：狗 / 猫 / 其他。
- 1-5 张宠物照片。
- 风格选择：像素风 / 卡通贴纸风 / 软萌 3D 风。

MVP 可用预置示例图或占位生成结果，不强依赖真实 AI 图像生成接口。

输出：

- 宠物电子形象。
- 角色卡：名字、性格、称号、今日口头禅。

#### 6.1.2 卡通家庭场景

提供默认房间模板：

- 客厅。
- 狗窝/猫窝。
- 饭盆区。
- 门口。
- 沙发。
- 阳台/窗边。

MVP 不支持复杂户型编辑，只支持选择预设场景和区域。

#### 6.1.3 宠物状态协议

系统内部使用统一状态结构：

| 字段 | 说明 | 示例 |
| --- | --- | --- |
| timestamp | 状态时间 | 2026-07-07 10:30 |
| source | 数据来源 | manual / mock_collar / imported_tracker |
| zone | 大概区域 | bed / bowl / door / sofa / window |
| activityLevel | 活动强度 | low / medium / high |
| confidence | 置信度 | 0.7 |
| eventHint | 事件暗示 | sleeping / waiting / eating / patrolling |
| moodHint | 可选情绪 | calm / excited / bored |

#### 6.1.4 桌宠状态演绎

状态映射规则：

- zone=bed + activityLevel=low -> 宠物在窝里睡觉。
- zone=door + activityLevel=medium -> 宠物在门口巡逻或等主人。
- zone=bowl + activityLevel=medium/high -> 宠物在饭盆附近转圈。
- zone=sofa + activityLevel=low -> 宠物趴在沙发休息。
- activityLevel=high -> 宠物跑动、跳跃或触发玩具动画。

MVP 允许状态演绎有随机变化，但必须保留“由数据驱动”的解释。

#### 6.1.5 用户互动

MVP 提供 4 个互动动作：

- 摸摸。
- 投喂。
- 叫名字。
- 逗玩具。

互动后宠物触发短动画和气泡文案。

#### 6.1.6 今日小剧场

系统根据一天中的状态序列生成：

- 今日时间线。
- 3-5 个剧情节点。
- 一段宠物第一视角文案。
- 一张可分享长图。

MVP 可先生成静态长图或页面，不强制生成视频。

#### 6.1.7 设备数据模拟器

由于第三方项圈 API 不一定开放，MVP 必须内置模拟器：

- 一键播放“上班日模式”。
- 一键播放“活跃日模式”。
- 一键播放“安静日模式”。
- 一键播放“门口等待模式”。

模拟器用于 Demo、测试和用户理解。

### 6.2 MVP 可选功能

- 15 秒短动画导出。
- 表情包生成。
- 桌面悬浮窗模式。
- 声音上传和拟人化台词生成。
- 第三方项圈数据 CSV/JSON 导入。

### 6.3 MVP 不做功能

- 实时绑定真实项圈 OAuth。
- 视频监控。
- 精准室内定位。
- 健康诊断。
- 多宠复杂关系。
- 用户社区。
- 商城闭环。

## 7. 数据接入策略

### 7.1 阶段一：模拟数据

目标：验证桌宠体验与用户价值。

数据来源：

- 系统内置模拟状态。
- 用户手动添加状态。
- 用户上传照片和事件描述。

优点：

- 不被第三方 API 卡住。
- 两天内可完成 Demo。
- 能验证核心体验是否打动人。

### 7.2 阶段二：文件导入

目标：接近真实设备数据。

支持：

- CSV。
- JSON。
- GPX/KML 轨迹文件。
- 手动粘贴第三方设备导出的活动记录。

系统将外部数据映射为统一状态协议。

### 7.3 阶段三：第三方集成

目标：与已有设备生态合作或适配。

优先级：

1. Home Assistant / Tractive 集成思路：Home Assistant 已有 Tractive 集成，可作为智能家居中间层参考。
2. Tractive：适合路径和位置历史。
3. Fi：适合活动、睡眠和行为状态。
4. FitBark：如果可获得 API，可用于活动和健康指数。
5. Furbo/Petcube：适合作为视频事件来源，但隐私与权限复杂，后置。

说明：第三方设备 API 状态、权限和稳定性需要单独确认。PawRoom 不应在 MVP 阶段承诺“实时接入所有项圈”。

## 8. 技术方案

### 8.1 推荐架构

前端桌面应用：

- Tauri 或 Electron。
- React / Vue 均可。
- 2D 场景使用 PixiJS 或 Phaser。
- 桌宠动画使用 Spine、Live2D、Sprite Sheet 或 Lottie。

后端/本地服务：

- MVP 可先做纯前端。
- 后续增加 Node.js / Python 服务做数据导入、AI 调用、状态推断。

AI 能力：

- 图像风格化：调用第三方图像生成/风格化模型。
- 文案生成：根据状态序列生成今日小剧场。
- 行为演绎：用规则引擎 + LLM 生成剧情节点。

### 8.2 核心模块

#### 模块 1：Pet Avatar Generator

职责：

- 接收宠物照片。
- 生成或选择电子宠物形象。
- 输出角色卡和基础动效资源。

MVP 实现：

- 用预置宠物素材或占位图模拟。
- 支持用户选择风格。

后续实现：

- 接入图像生成模型。
- 做同一宠物跨表情、动作的一致性生成。

#### 模块 2：Room Scene Engine

职责：

- 渲染卡通家居地图。
- 管理区域、宠物位置、场景物件。
- 展示宠物移动和状态变化。

MVP 实现：

- 预设 1-2 个房间模板。
- 支持 5-6 个互动区域。

#### 模块 3：Pet State Adapter

职责：

- 把手动输入、模拟数据、第三方设备数据转换成统一状态协议。
- 隔离不同设备的数据差异。

MVP 实现：

- Mock Adapter。
- Manual Adapter。
- File Import Adapter。

后续实现：

- Tractive Adapter。
- Fi Adapter。
- Home Assistant Adapter。

#### 模块 4：Behavior Director

职责：

- 根据宠物状态决定动画、位置、气泡文案和小剧场剧情。
- 控制“大概真实 + 可爱演绎”的边界。

MVP 实现：

- 规则引擎。
- 少量随机变体。

后续实现：

- LLM 生成剧情节点。
- 用户可训练宠物性格。

#### 模块 5：Interaction Layer

职责：

- 处理摸摸、投喂、叫名字、逗玩具等互动。
- 触发动作、反馈、音效、情绪变化。

#### 模块 6：Memory Exporter

职责：

- 输出今日小剧场。
- 输出长图、短视频、表情包、角色卡。

MVP 实现：

- 导出静态图片或 HTML 页面截图。

后续实现：

- 导出 MP4/GIF。
- 接入实体周边下单。

- 适合做技术验证和合规参考。

注意点：

- 官方 SDK 和模型有授权边界，商业化前必须确认 License。

#### 8.3.4 phaserjs/phaser

地址：https://github.com/phaserjs/phaser

用途：

- 2D 游戏引擎。
- 可用于房间地图、小游戏、宠物移动、交互事件。

适配价值：

- 适合做“宠物小世界 + 轻小游戏”。
- 比普通 DOM 动画更适合游戏化交互。

注意点：

- 桌面悬浮窗需要搭配 Electron/Tauri。
- UI 与游戏场景之间需要清晰分层。

#### 8.3.5 pixijs/pixijs

地址：https://github.com/pixijs/pixijs

用途：

- 2D WebGL 渲染引擎。
- 适合高性能桌宠动画和房间场景。

适配价值：

- 适合 MVP 和后续商业版。
- 可与 pixi-live2d-display 组合。

注意点：

- 游戏逻辑需要自行设计。

#### 8.3.6 tauri-apps/tauri

地址：https://github.com/tauri-apps/tauri

用途：

- 跨平台桌面应用框架。

适配价值：

- 适合做轻量桌面常驻窗口。
- 体积通常比 Electron 小。

注意点：

- 前端开发体验好，但系统级悬浮窗、置顶、透明窗口需要专项验证。

#### 8.3.7 electron/electron

地址：https://github.com/electron/electron

用途：

- 跨平台桌面应用框架。

适配价值：

- 生态成熟，适合快速做桌面窗口、透明窗、托盘、自动更新。

注意点：

- 应用体积和资源占用较高。

#### 8.3.8 Home Assistant Tractive 集成

文档：https://www.home-assistant.io/integrations/tractive/

用途：

- 通过 Home Assistant 获取 Tractive 设备相关状态。

适配价值：

- 可以作为后续真实设备数据接入的低成本路径。
- 先接 Home Assistant，再由 PawRoom 读取标准化状态，降低直接对接硬件厂商 API 的难度。

注意点：

- 面向智能家居用户，普通用户配置门槛较高。
- 不适合作为大众版首发路径。

### 8.4 推荐二改路线

#### 2 天 Demo 路线

- 用 React + PixiJS 或 Phaser 做网页原型。
- 使用预置宠物素材模拟卡通化结果。
- 使用 Mock Adapter 播放宠物一天状态。
- 输出桌面场景、小剧场和互动。

#### 2 周 MVP 路线

- 用 Tauri 或 Electron 包装成桌面应用。
- 引入 PixiJS / Phaser 场景。
- 增加文件导入和模拟数据编辑。
- 增加图片导出。
- 增加 1 个真实用户照片生成流程，哪怕先半自动。

#### 后续真实数据路线

- 优先调研 Home Assistant Tractive 集成可行性。
- 若有稳定数据源，再开发 Device Adapter。
- 不把第三方实时接入作为 MVP 成败条件。

## 9. 交互流程

### 9.1 首次使用流程

1. 用户打开 PawRoom。
2. 选择“创建我的宠物小世界”。
3. 上传宠物照片或选择示例宠物。
4. 输入宠物名字。
5. 选择视觉风格。
6. 选择家庭场景模板。
7. 选择是否使用模拟项链数据。
8. 进入桌面小世界。

### 9.2 上班桌面流程

1. 用户启动桌宠窗口。
2. 系统读取当前宠物状态。
3. 宠物出现在对应区域。
4. 用户可随时点击互动。
5. 系统自动展示状态气泡。
6. 桌宠低频更新，避免打扰工作。

### 9.3 今日回放流程

1. 用户点击“今日小剧场”。
2. 系统读取当天状态序列。
3. 生成 3-5 个剧情节点。
4. 展示卡通时间线。
5. 用户导出图片或分享。

## 10. 功能需求明细

### 10.1 宠物创建页

优先级：P0

需求：

- 支持输入宠物名称。
- 支持选择宠物类型。
- 支持上传图片或选择示例宠物。
- 支持选择视觉风格。
- 支持生成角色卡。

验收标准：

- 用户能在 1 分钟内完成宠物创建。
- 即使未接入 AI 图片接口，也能看到清晰的卡通宠物占位结果。

### 10.2 房间场景页

优先级：P0

需求：

- 展示卡通房间。
- 展示宠物当前位置和动画。
- 展示当前状态标签。
- 支持切换模拟数据场景。

验收标准：

- 用户能一眼理解宠物在“家里的哪个区域”。
- 不同状态有可感知的视觉变化。

### 10.3 状态数据模拟器

优先级：P0

需求：

- 提供至少 4 套模拟状态。
- 支持播放/暂停。
- 支持跳转到某个时间点。
- 支持展示状态来源。

验收标准：

- 开发和演示时无需真实项圈也能完整跑通体验。

### 10.4 互动动作

优先级：P0

需求：

- 摸摸。
- 投喂。
- 叫名字。
- 逗玩具。

验收标准：

- 每个动作都有动画反馈和文案反馈。
- 互动不影响基础状态播放。

### 10.5 今日小剧场

优先级：P0

需求：

- 自动提取当天关键状态。
- 生成宠物第一视角文案。
- 生成可分享页面或长图。

验收标准：

- 小剧场内容能对应当天状态。
- 用户能理解“真实数据 + 可爱演绎”的关系。

### 10.6 数据导入

优先级：P1

需求：

- 支持 JSON 导入。
- 支持 CSV 导入。
- 支持导入后预览状态序列。
- 支持错误提示。

验收标准：

- 示例文件可成功映射为统一状态协议。

### 10.7 桌面化

优先级：P1

需求：

- 支持桌面窗口。
- 支持置顶/取消置顶。
- 支持窗口大小调整。
- 支持低干扰模式。

验收标准：

- 用户可以在上班时把窗口放在屏幕角落。

## 11. 非功能需求

### 11.1 性能

- 桌面场景空闲时 CPU 占用应尽量低。
- 动画帧率目标 30 FPS 以上。
- 场景加载时间小于 3 秒。

### 11.2 隐私

- MVP 默认本地处理模拟数据和上传素材。
- 不默认开启摄像头。
- 不采集真实家庭视频。
- 用户导入设备数据前必须说明用途。

### 11.3 可靠性

- 没有设备数据时，产品仍可通过手动/模拟数据运行。
- 第三方数据缺失时，桌宠应降级为装饰和互动模式。

### 11.4 合规与伦理

- 不宣称医疗诊断能力。
- 不使用“宠物真实想法翻译”等误导表达。
- 明确小剧场是 AI 拟人化创作。
- 使用第三方开源项目时必须遵守 License。

## 14. 测试方案

### 14.1 单元测试

重点测试：

- Pet State Adapter 状态映射。
- Behavior Director 状态到动作的映射。
- Memory Exporter 对状态序列的剧情提取。

原则：

- 测外部行为，不测内部实现。
- 使用固定模拟数据验证输出稳定性。

### 14.2 集成测试

场景：

- 创建宠物 -> 进入房间 -> 播放模拟数据 -> 触发互动 -> 生成小剧场。
- 导入 JSON -> 映射状态 -> 场景渲染。
- 无数据 -> 降级为装饰桌宠。

### 14.3 用户测试

测试对象：

- 5 位上班族养宠人。
- 3 位高分享欲养宠人。
- 2 位已有智能项圈或宠物摄像头用户。

测试问题：

- 你是否理解这个产品不是监控，而是可爱演绎？
- 你是否愿意上班时开着它？
- 你是否愿意上传自己宠物的照片？
- 你是否愿意为高清小剧场或表情包付费？
- 哪些状态你最想看到？

## 16. 风险与应对

### 风险 1：用户认为它不够真实

应对：

- 明确产品是“AI 演绎”，不是精准监控。
- 保留状态来源和置信度。
- 给用户手动修正和添加动作的能力。

### 风险 2：第三方设备 API 不开放

应对：

- MVP 不依赖真实 API。
- 使用模拟数据和文件导入验证体验。
- 后续优先通过 Home Assistant 等中间层接入。

### 风险 3：产品变成普通 AI 宠物头像

应对：

- 强调持续存在的电子宠物分身。
- 强调状态驱动场景。
- 强调今日小剧场和互动。

### 风险 4：桌宠使用频率不够

应对：

- 降低打开成本。
- 做低干扰桌面常驻模式。
- 增加每日回放和分享价值。

### 风险 5：硬件竞品太强

应对：

- 不进入硬件定位赛道。
- 把 Tractive、Fi 等定位为数据来源，而非直接竞品。
- 产品壁垒放在视觉演绎和情绪体验。

## 18. 最终建议

PawRoom 的首版不要做硬件，也不要承诺精准实时监控。最优策略是：

1. 用模拟数据验证“上班时看卡通宠物小世界”是否有吸引力。
2. 用用户真实照片验证“专属宠物分身”是否比普通桌宠更有情绪价值。
3. 用今日小剧场验证用户是否愿意分享和付费。
4. 等体验验证成立后，再接入第三方项圈、Home Assistant 或文件导入数据。

推荐 MVP 口号：

> 上班时，在电脑桌面看见卡通版的它，正在家里的小世界里生活。
