## 1. 摘要

PawRoom 是一个“宠物项圈 + AI 监测定位系统平台”的概念验证产品。产品通过宠物项圈采集活动、位置区域、事件和设备状态，再由 AI 平台把这些数据演绎成一个可互动、可观看、可分享的桌面宠物小世界。

本阶段不制作实体硬件。项圈硬件只作为 Demo 支线，用来展示数据采集入口和平台联动逻辑。核心验证对象不是硬件量产能力，而是用户是否认可：

- 上班时在电脑桌面看到自家宠物的卡通小世界。
- 宠物项圈数据可以被转化为更有趣、更美观、更有陪伴感的桌面体验。
- AI 能把照片、活动、区域、声音和事件变成宠物电子分身、今日小剧场、互动动作和分享内容。

核心一句话：

> PawRoom 通过宠物项圈采集活动与位置状态，再用 AI 把这些数据演绎成一个可互动的桌面宠物小世界，让上班的人也能看到卡通版的它正在家里生活。

## 2. 背景与真实资料依据

### 2.1 现有竞品格局

主流宠物科技产品已经验证了“宠物远程关注”和“宠物数据采集”是真需求：

- Tractive：GPS 定位、实时追踪、虚拟围栏、位置历史、活动和睡眠数据。
- Fi：智能狗项圈，覆盖 GPS、活动、睡眠，Series 3 Plus 增加叫声、舔舐、抓挠、吃喝等行为检测。
- PetPace：偏健康监测，关注体温、脉搏、呼吸等生命体征。
- Invoxia Minitailz：宠物 GPS、活动、心率、呼吸等监测方向。
- Furbo / Petcube：宠物摄像头、远程查看、AI 提醒和互动。

这些竞品的强项是“采集和展示真实数据”，弱点是体验偏工具化：

- 地图、曲线、提醒对用户有用，但不够有情绪价值。
- 摄像头能看真实画面，但偏监控，隐私压力高。
- 智能项圈知道宠物状态，但缺少好玩、美观、可分享的表达方式。

PawRoom 不正面挑战这些硬件产品，而是做它们上层的“AI 演绎与陪伴平台”。

### 2.2 技术现实

本阶段项圈只做 Demo，不做实体硬件。真实硬件技术上可以有几类路线：

- 加速度计：判断静止、走动、奔跑、异常高活动。
- BLE 蓝牙信标：判断宠物大概靠近哪个区域，如狗窝、饭盆、门口。
- GPS：适合户外散步轨迹，不适合精准室内定位。
- UWB：可做更精准室内定位，但需要基站，成本和部署门槛较高。
- 麦克风：可识别叫声等声音事件，但涉及隐私、功耗和误识别。

MVP 不依赖真实项圈 API。用模拟项圈数据和可视化数据流完成体验验证。

## 3. 机会判断

### 3.1 目标用户是谁

#### 用户 A：上班族养宠人

- 白天宠物独自在家。
- 想知道宠物大概在干什么。
- 不想一直打开摄像头监控。
- 需要低干扰的陪伴感。

#### 用户 B：高分享欲宠物用户

- 经常发小红书、朋友圈、B站或抖音。
- 已经拍了大量宠物照片和视频。
- 希望内容更有趣，而不是普通照片堆叠。

#### 用户 C：已有宠物设备用户

- 已经购买宠物摄像头、智能项圈、定位器。
- 证明愿意为宠物科技产品付费。
- 对现有 App 的图表和提醒感到不够有趣。

#### 用户 D：新宠主人

- 刚开始养宠，拍照和记录频率高。
- 愿意尝试新鲜玩法。
- 对“我的宠物变成电子分身”有天然兴趣。

### 3.2 痛点是什么

核心痛点：

- 想知道宠物在家大概状态，但不想被监控画面绑住。
- 现有项圈数据过于工具化，缺少情绪价值和美感。
- 宠物照片、视频很多，但只是静态存档，很难变成持续互动。
- 普通桌宠和虚拟宠物不是“我的宠物”，缺少个体情感连接。
- 用户在上班时需要低干扰的安心感和陪伴感。
- 用户想分享宠物日常，但普通素材同质化严重。

### 3.3 为什么现在值得做

1. 硬件侧需求已经被验证  
   Tractive、Fi、PetPace、Furbo、Petcube 等产品证明用户愿意为宠物定位、监测、远程查看付费。

2. 生成式 AI 让宠物内容转译成本下降  
   过去要把宠物照片变成角色、动效、小剧场，需要设计师和动画师；现在可以通过 AI 半自动生成。

3. 用户对宠物拟人化和情绪消费接受度高  
   宠物画像、定制周边、纪念册、表情包、桌宠都属于已存在的消费方向。

4. 现有硬件缺少“表达层”  
   项圈知道宠物在动，但没有把“动”变成“它今天在家经历了什么”。

5. 可以低成本验证  
   不需要先做实体项圈。用模拟数据、可视化项圈 Demo 和桌面平台即可验证核心体验。

### 3.4 机会边界

不进入：

- 精准 GPS 项圈竞争。
- 医疗级健康监测。
- 24 小时视频监控。
- 实体硬件量产。

切入：

- 宠物状态 AI 理解。
- 桌面小世界。
- 宠物电子分身。
- 状态驱动动效。
- 今日小剧场。
- 可分享内容和宠物周边入口。

## 4. 产品定位

### 4.1 产品定位

PawRoom 是一个宠物项圈数据驱动的 AI 桌面宠物小世界。

它不是严肃监控工具，而是把宠物活动数据转化为可爱、低干扰、可互动的桌面陪伴体验。

### 4.2 用户价值

- 大概知道宠物在家状态。
- 上班时获得陪伴感。
- 把自己的宠物变成电子分身。
- 把冷冰冰的数据变成小剧场。
- 生成表情包、长图、短动画等可分享内容。

### 4.3 差异化

| 类型 | 典型产品 | 核心价值 | PawRoom 差异化 |
| --- | --- | --- | --- |
| GPS 项圈 | Tractive、Fi | 定位和轨迹 | 把状态转成桌面小世界 |
| 健康项圈 | PetPace、Invoxia | 生命体征 | 不做医疗，只做非医疗状态演绎 |
| 摄像头 | Furbo、Petcube | 真实画面 | 不展示家中真实视频，降低隐私压力 |
| AI 头像 | 图像生成工具 | 单张风格化图片 | 生成持续存在的宠物分身 |
| 普通桌宠 | Live2D/桌宠工具 | 虚拟角色陪伴 | 角色来自真实宠物和状态数据 |

## 5. AI 技术如何提升交互和逻辑

### 5.1 从数据展示到状态理解

传统设备展示的是活动量、位置、轨迹。AI 层把数据转成用户更容易理解的状态：

- 活动低 + 狗窝区域 -> 可能在睡觉。
- 门口区域 + 中等活动 -> 可能在等主人或巡逻。
- 饭盆区域 + 中等活动 -> 可能在吃饭或找吃的。
- 活动高 + 客厅区域 -> 可能在玩耍。

AI 的价值是把硬件数据翻译成“宠物在做什么”。

### 5.2 从监控视角到陪伴视角

摄像头是“看见真实画面”，PawRoom 是“看见宠物小世界”。  
用户不是打开监控，而是在桌面上看到一只卡通版的自家宠物。

这降低隐私压力，也更适合上班场景。

### 5.3 从静态照片到动态角色

用户上传照片后，AI 生成：

- 宠物卡通形象。
- 角色性格。
- 表情包。
- 动作风格。
- 今日口头禅。
- 小剧场旁白。

照片不再只是相册素材，而是角色资产。

### 5.4 从路径数据到卡通地图

AI 把家中区域游戏化：

- 狗窝 = 充电基地。
- 饭盆 = 能量补给站。
- 门口 = 等待峡谷。
- 沙发 = 午睡山丘。
- 窗边 = 侦察塔。

用户看到的不是坐标，而是宠物今天在小世界里的行动路线。

### 5.5 从固定桌宠到状态驱动桌宠

普通桌宠动作是循环的。PawRoom 桌宠由状态驱动：

- 今天安静 -> 趴着、睡觉、偶尔抬头。
- 今天活跃 -> 跑动、追玩具、跳跃。
- 靠近门口 -> 等主人、巡逻。
- 用户点击互动 -> 根据当前状态生成不同反馈。

### 5.6 从手动记录到自动小剧场

用户不需要写宠物日记。AI 根据状态序列生成：

- 今日关键节点。
- 宠物第一视角文案。
- 四格漫画脚本。
- 15 秒短动画脚本。
- 可分享长图。

## 6. MVP 范围

### 6.1 必做功能

#### 6.1.1 宠物创建

输入：

- 宠物名称。
- 宠物类型。
- 1-5 张宠物照片。
- 风格选择：像素风 / 卡通贴纸风 / 软萌 3D 风。

输出：

- 宠物电子形象。
- 宠物角色卡。

MVP 可使用预置素材模拟 AI 生成效果。

#### 6.1.2 项圈硬件 Demo

功能：

- 展示项圈概念图。
- 展示设备 ID、电量、连接状态。
- 展示实时模拟数据流。
- 展示项圈采集能力：活动强度、大概区域、事件提示、置信度。

注意：该项圈不需要实体物，只做 UI/动效 Demo。

#### 6.1.3 卡通家居地图

默认区域：

- 狗窝/猫窝。
- 饭盆。
- 门口。
- 沙发。
- 窗边。
- 玩具区。

#### 6.1.4 数据协议

项圈 Demo 向平台发送模拟数据包：

```json
{
  "deviceId": "collar_demo_001",
  "petId": "pet_001",
  "timestamp": "2026-07-07T10:30:00+08:00",
  "activityLevel": "medium",
  "zone": "door",
  "motionHint": "pacing",
  "eventHint": "waiting",
  "battery": 82,
  "confidence": 0.72,
  "source": "simulated_collar"
}
```

#### 6.1.5 桌面小世界

能力：

- 宠物在房间中移动。
- 根据状态切换动作。
- 显示状态气泡。
- 支持低干扰桌面窗口。

#### 6.1.6 互动动作

MVP 至少支持：

- 摸摸。
- 投喂。
- 叫名字。
- 逗玩具。

#### 6.1.7 今日小剧场

根据状态序列生成：

- 时间线。
- 3-5 个剧情节点。
- 宠物第一视角旁白。
- 可分享长图。

### 6.2 可选功能

- 15 秒短视频导出。
- 表情包生成。
- 声音上传和台词生成。
- JSON/CSV 文件导入。
- 桌面悬浮窗模式。

### 6.3 不做功能

- 不制作实体项圈。
- 不接入真实实时 API。
- 不做医疗诊断。
- 不做精准室内定位。
- 不做完整电商系统。
- 不做复杂多人社区。

## 7. 核心用户流程

### 7.1 首次创建

1. 用户打开 PawRoom。
2. 上传宠物照片。
3. 输入宠物名称。
4. 选择卡通风格。
5. 生成宠物电子分身。
6. 进入桌面小世界。

### 7.2 项圈数据演示

1. 用户进入“项圈连接”页。
2. 看到项圈概念图和连接状态。
3. 点击“开始模拟采集”。
4. 数据流实时进入平台。
5. 桌面场景中的宠物开始根据数据变化。

### 7.3 上班桌面陪伴

1. 用户把 PawRoom 放在电脑角落。
2. 宠物在卡通家居地图中活动。
3. 用户偶尔点击互动。
4. 系统低频显示宠物状态气泡。

### 7.4 下班回看

1. 用户点击“今日小剧场”。
2. 系统生成今天的宠物行动故事。
3. 用户导出长图或分享。

## 8. 技术方案

### 8.1 架构

前端：

- React / Vue。
- PixiJS 或 Phaser 渲染 2D 房间和宠物动画。
- Tauri 或 Electron 封装桌面端。

AI 层：

- 图片风格化生成宠物形象。
- LLM 生成状态解释和小剧场文案。
- 规则引擎做状态到动作的稳定映射。

数据层：

- Mock Collar Adapter。
- Manual State Adapter。
- File Import Adapter。
- 后续 Third-party Device Adapter。

### 8.2 核心模块

#### Pet Avatar Generator

负责把宠物照片转为电子宠物形象。MVP 可用预置素材替代真实生成。

#### Collar Demo Simulator

负责模拟项圈数据，展示数据采集和实时传输。

#### Pet State Adapter

负责把不同来源的数据转换成统一状态协议。

#### Behavior Director

负责把状态映射为动画、文案和剧情。

#### Room Scene Engine

负责渲染桌面房间、宠物移动和互动。

#### Memory Exporter

负责输出今日小剧场、长图、表情包。

## 9. 商业判断

### 9.1 付费用户

1. 上班族养宠人  
   为陪伴感和轻度安心付费。

2. 宠物内容分享用户  
   为表情包、小剧场、长图和短视频付费。

3. 已有智能设备用户  
   为“把已有硬件数据变有趣”付费。

4. 宠物纪念和周边消费用户  
   为立牌、贴纸、钥匙扣、纪念册付费。

### 9.2 为什么付费

用户付费的不是监测数据本身，而是：

- 自家宠物变成电子分身。
- 上班时可爱的桌面陪伴。
- 今天在家的卡通小剧场。
- 可分享的宠物内容。
- 可实体化的宠物 IP 周边。

### 9.3 定价

### 9.4 获客

## 10. 2 周验证计划

### 10.1 验证目标

验证 4 个问题：

1. 用户是否理解 PawRoom 不是监控，而是 AI 桌面宠物小世界。
2. 用户是否愿意上传自家宠物照片。
3. 用户是否愿意上班时常驻打开桌宠。
4. 用户是否愿意为小剧场、表情包、动作包或周边付费。

### 10.2 第 1-3 天：需求验证

任务：

- 访谈 8-10 位养宠用户。
- 找 3 位已有宠物摄像头或智能项圈用户。
- 收集小红书、B站、知乎关于“宠物独自在家”“上班想狗”“宠物摄像头”“宠物项圈”的评论。
- 梳理 Tractive、Fi、Furbo、Petcube 的卖点和用户吐槽。

产出：

- 10 条真实痛点证据。
- 3 个用户画像。
- 竞品差异化表。

### 10.3 第 4-6 天：Demo 验证

任务：

- 做桌面小世界低保真原型。
- 做项圈 Demo 数据面板。
- 用模拟数据驱动宠物在房间中移动。
- 准备 3 个场景：安静日、活跃日、等主人日。

产出：

- 可点击原型。
- 15 秒演示视频。
- 项圈数据流转图。

### 10.4 第 7-9 天：真实素材测试

任务：

- 邀请 5 位用户上传宠物照片。
- 半自动生成卡通宠物形象。
- 让用户选择家庭区域和宠物动作。
- 给每位用户生成一份今日小剧场。

验证：

- 形象是否有情绪价值。
- 桌宠是否愿意常驻。
- 小剧场是否愿意分享。

### 10.5 第 10-12 天：付费测试

任务：

- 测试 9.9、19.9、59 元三个价格锚点。
- 测试高清导出、表情包、实体周边意向。
- 使用问卷、预约链接或微信收款码测试真实支付意愿。

通过标准：

- 至少 3 位用户愿意支付 9.9 元以上。
- 至少 2 位用户愿意为实体周边留下联系方式。
- 至少 3 位用户表示愿意上班时打开桌宠。

### 10.6 第 13-14 天：复盘决策

根据反馈做决策：

- 如果用户只喜欢生成图，不关心桌宠：转向宠物 IP 内容工具。
- 如果用户喜欢桌宠但不关心项圈：先做纯软件桌宠。
- 如果用户关心在家状态：继续推进第三方项圈数据接入。
- 如果用户愿意为周边付费：加入周边商业化。

## 11. 测试与验收

### 11.1 功能验收

- 用户能创建宠物电子分身。
- 项圈 Demo 能持续输出模拟数据。
- 桌面小世界能响应数据变化。
- 用户能进行至少 4 个互动动作。
- 系统能生成今日小剧场。

### 11.2 体验验收

- 用户能理解“数据驱动 + AI 演绎”。
- 用户不会误以为产品是精准监控。
- 桌面场景美观、低干扰。
- 用户愿意截图或分享。

### 11.3 商业验收

- 至少 3 位用户愿意为导出或表情包付费。
- 至少 2 位用户表达实体周边兴趣。
- 至少 3 位用户愿意上班期间打开桌宠。

## 12. 风险与应对

### 12.1 用户认为不真实

应对：

- 明确展示置信度。
- 表达为“可爱演绎”而非“精准监控”。
- 允许用户手动补充动作和事件。

### 12.2 硬件叙事过重

应对：

- 明确项圈只是 Demo 支线。
- 不承诺硬件量产。
- 核心壁垒放在 AI 平台。

### 12.3 第三方 API 不开放

应对：

- MVP 使用模拟数据。
- 支持文件导入。
- 后续通过 Home Assistant 等中间层探索接入。

### 12.4 用户只玩一次

应对：

- 增加桌面常驻低干扰体验。
- 增加每日小剧场。
- 增加表情包和周边导出。
